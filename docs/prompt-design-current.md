# 当前 Prompt 设计说明

文档目的：记录当前已经实现的提示词设计意图、职责边界、输入输出约束和已知失败模式，便于后续继续调 prompt，而不是每次从代码里反推。

关联代码：

- [app/prompts.py](/Users/test/code/aipm_resume_analyzer/app/prompts.py)
- [app/llm_client.py](/Users/test/code/aipm_resume_analyzer/app/llm_client.py)

## 1. 当前 Prompt 的定位

当前 prompt 只服务一个任务：

- 在规则分析结果基础上，生成更自然、更具体的中文结果文案

它不负责：

- 判断最终 `recommendation`
- 重算 `match_score`
- 改写 `job_type`
- 从零做岗位匹配决策

结论：

- 当前 LLM 是 `结果增强层`
- 当前规则引擎是 `决策主链路`

## 2. 为什么这样设计

核心原因有 4 个：

- `可控`
  最终投递建议仍由规则决定，避免模型自由发挥导致结果漂移。

- `可解释`
  模型只围绕规则结果和原始输入写解释，边界更清楚。

- `可回退`
  如果模型超时、返回脏 JSON、或者 API 不可用，后端可以直接退回纯规则结果。

- `便于调试`
  当结果不理想时，可以先判断是规则问题，还是 prompt 写法问题，而不是两者混在一起。

## 3. 当前 Prompt 结构

当前只有一组 prompt，分为两部分：

1. `system prompt`
2. `user prompt`

代码位置在 [app/prompts.py](/Users/test/code/aipm_resume_analyzer/app/prompts.py)。

### 3.1 System Prompt 的职责

当前 `LLM_RESULT_SYSTEM_PROMPT` 的作用是给模型划硬边界。

它主要约束 5 件事：

- 模型身份：`AI PM 求职判断助手`
- 数据边界：只能基于 `JD + 简历 + 规则结果`
- 决策边界：不能改 `recommendation / match_score / job_type`
- 事实边界：不能编造经历
- 格式边界：必须只输出 JSON

这些约束的目的分别是：

- 防止模型开始聊天、解释、安慰用户
- 防止模型绕过规则直接下新结论
- 防止模型为了写得更像人话而虚构候选人经历
- 防止输出 markdown、段落、标题，导致解析失败

### 3.2 User Prompt 的职责

当前 `build_llm_result_user_prompt(payload)` 的作用是把任务、输出结构和输入数据一起喂给模型。

它主要包含 3 层信息：

- 任务目标：
  让模型把现有结果写得更自然、更具体，但仍然克制

- 输出约束：
  只返回 `summary, strengths, risks, next_actions`

- 风格约束：
  - `summary` 必须是一句中文结论
  - 其余 3 个字段必须是 `2-4` 条中文短句数组
  - 尽量引用 JD 或简历里的真实线索
  - 不要输出空字段

最后会把完整 `payload` JSON 直接附在 prompt 后面。

## 4. 当前输入设计

当前 user prompt 输入的是一个结构化 `payload`，字段包括：

```json
{
  "user_level": "转岗PM",
  "goal": "转AI",
  "jd_text": "原始JD文本",
  "resume_text": "原始简历文本",
  "rule_result": {
    "recommendation": "可投",
    "match_score": 68,
    "job_type": "真实AI PM",
    "job_signals": {},
    "candidate_signals": {},
    "strengths": [],
    "risks": [],
    "next_actions": [],
    "summary": "规则生成的一句总结"
  }
}
```

这样设计的目的：

- `jd_text`
  给模型保留原始岗位上下文，方便写得更具体

- `resume_text`
  让模型引用候选人真实经历，而不是只盯着规则结果

- `rule_result`
  给模型一个已经定好的判断框架，避免它重新做决策

- `user_level` 和 `goal`
  给模型一点上下文，帮助文案更贴用户状态

## 5. 当前输出设计

模型被要求只返回 4 个字段：

- `summary`
- `strengths`
- `risks`
- `next_actions`

原因很直接：

- 这 4 个字段是前端当前真正展示的结果骨架
- 它们是最适合让模型增强的部分
- 这些字段改写后，即使失败，也容易 fallback 到规则结果

当前刻意不让模型返回：

- `recommendation`
- `match_score`
- `job_type`
- 新增的自定义字段

这是为了保证后端合并逻辑简单，减少 schema 漂移。

## 6. 当前设计在防什么问题

这套 prompt 不是为了“写得华丽”，而是在防 5 类常见问题。

### 6.1 防模型改判

最危险的问题是：

- 规则说“谨慎”
- 模型因为语气自由发挥，写成了“可以大胆投”

所以 prompt 明确写死：

- 不能改 `recommendation`
- 不能改 `match_score`
- 不能改 `job_type`

### 6.2 防模型编造证据

求职场景里，编造经历是高风险错误。

所以 prompt 明确要求：

- 只能根据 JD、简历和规则结果生成解释
- 不能编造不存在的项目经历

### 6.3 防输出不可解析

如果模型返回：

- markdown 列表
- 段落解释
- 额外标题
- 代码块外混杂说明文字

后端 JSON 提取就容易失败。

所以 prompt 明确要求：

- 只输出 JSON
- 不要输出 markdown
- 不要解释

### 6.4 防输出太空

模型如果只写泛话，比如：

- “你有一定优势”
- “建议继续努力”

对产品没有价值。

所以 user prompt 明确要求：

- 尽量引用 JD 或简历真实线索
- 输出要更自然、更具体

### 6.5 防输出太散

如果不限制字段，模型很容易自己加：

- `analysis`
- `confidence`
- `notes`
- `improvement_plan`

这会让后端 schema 和前端渲染都变复杂。

所以当前只允许固定 4 个字段。

## 7. 当前已知不足

这版 prompt 能跑，但还比较粗。

当前主要问题有 6 个：

- `没有强制证据引用格式`
  现在只是“尽量引用”，不是硬约束，所以模型仍可能写泛泛总结。

- `没有对风格做更细限制`
  现在只有“自然、具体、克制”，还没有定义更细的语气边界。

- `没有对长短做更精确控制`
  虽然限制了字段和数组数量，但没有限制每条句子的长度。

- `没有对风险优先级做约束`
  模型可能把次要问题写得比主要短板更重。

- `没有显式处理信息冲突`
  如果 `resume_text` 和 `rule_result` 给出的信号不一致，prompt 里没有专门说明如何处理。

- `没有拆成抽取和生成两步`
  当前模型同时看原始文本和规则结果，未来复杂度上来后可能会互相干扰。

## 8. 当前失败模式

后续调 prompt 时，优先盯这几类失败。

### 8.1 输出不是合法 JSON

表现：

- 多了说明文字
- 多了 markdown 包裹
- JSON 缺括号

结果：

- `_extract_json()` 失败
- 后端直接 fallback 到纯规则结果

### 8.2 文案太泛

表现：

- 看起来像通用求职建议
- 没有引用 JD 或简历中的具体线索

结果：

- 用户会觉得“LLM 说了等于没说”

### 8.3 文案越权

表现：

- 明面上没改 `recommendation`
- 但文案语气已经在暗示相反结论

结果：

- 产品结论和解释层冲突

### 8.4 编造经历

表现：

- 把用户的“做过原型”写成“有完整落地经验”
- 把“会基础 API/SQL”写成“具备较强技术能力”

结果：

- 分析可信度下降

### 8.5 句子过长

表现：

- 一条 `strength` 或 `risk` 变成一长段

结果：

- 前端可读性差
- 卡片展示拥挤

## 9. 当前调优优先级

如果后面继续调 prompt，建议按这个顺序来：

1. `先提升具体性`
   让模型更稳定地引用 JD / 简历证据

2. `再收紧语气边界`
   保证解释层不会和 recommendation 打架

3. `再控制输出长度`
   让前端结果更干净

4. `最后再考虑拆分多步 prompt`
   只有在当前单步方案明显不够时再做

## 10. 一句话总结

当前 prompt 的本质不是“让模型更聪明”，而是：

- 在规则结论不变的前提下
- 让结果更像人话
- 同时尽量不失控
