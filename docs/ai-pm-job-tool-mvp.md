# AI PM 岗位判断工具 MVP 文档

日期：2026-06-17  
项目阶段：MVP 定义  
项目性质：单人开发，可独立发布的轻量 Web 工具

## 1. 项目一句话

一个面向 AI 产品经理求职者的岗位判断工具。  
用户输入 `JD + 简历 + 当前背景`，系统输出 `这岗值不值得投`、`你匹配在哪里`、`风险在哪里`、`下一步怎么做`。

## 2. MVP 目标

### 2.1 目标

MVP 只验证一件事：

`用户是否愿意用一个工具，先判断 AI PM 岗位值不值得投，再决定怎么投。`

### 2.2 不追求的目标

MVP 不追求：

- 不做完整求职平台
- 不做招聘聚合站
- 不做自动海投
- 不做完整简历优化器
- 不做面试题库
- 不做社交或社区

### 2.3 MVP 成功标准

MVP 成功的标准不是功能多，而是以下 4 点成立：

1. 用户能在 `3 分钟内` 完成一次分析
2. 用户能得到明确的 `投递判断`
3. 用户觉得输出 `像真的懂 AI PM 岗位`
4. 用户愿意把结果发给朋友，或拿去继续改简历/准备面试

## 3. 用户

### 3.1 核心用户

目标用户是正在投递 `AI 产品经理` 岗位的人。

重点包括：

- 想从传统 PM 转 AI PM 的人
- 已有 PM 经验，但看不懂 AI PM JD 的人
- 有一些 AI/Agent/Prompt/工作流实践，但不确定够不够投的人
- 已经在投岗位，但无法判断岗位真实含金量的人

### 3.2 非目标用户

MVP 不服务：

- 泛求职用户
- 纯研发岗位求职者
- 只想做 ATS 简历优化的人
- 只想生成简历模板、不关心岗位判断的人

### 3.3 用户分层

MVP 里先支持 4 类用户身份：

- `新人`
- `转岗 PM`
- `有经验 PM`
- `有 AI 项目经验`

这个身份不是做复杂画像，而是帮助调整结果解释的语气和权重。

## 4. 场景

### 4.1 核心场景

用户在看到一个 AI PM JD 后，脑子里会立刻出现 4 个问题：

1. `我配不配投`
2. `这个岗位到底真不真`
3. `我缺的到底是什么`
4. `如果要投，我现在该怎么改`

MVP 就是为这个时刻服务。

### 4.2 使用时机

最典型的使用时机有 3 个：

1. `投递前`
   用户先判断值不值得投，避免盲投。

2. `改简历前`
   用户先知道岗位看重什么，再决定改哪里。

3. `面试准备前`
   用户快速理解这个岗位真正关心的能力。

### 4.3 用户当前替代方案

用户在这类求职场景下通常会这么做：

- 自己硬读 JD
- 去问朋友或群友
- 搜类似岗位经验帖
- 用 ATS 工具看关键词匹配
- 靠感觉判断

问题是：这些方法要么效率低，要么只解决“字面匹配”，不解决“岗位判断”。

## 5. 用户需求

### 5.1 核心需求

MVP 要满足 5 个核心需求：

1. `岗位判断`
   这个岗位值得投吗？

2. `匹配判断`
   我的经历和这个岗位有哪些对得上，哪些对不上？

3. `岗位类型识别`
   这是成长型岗位、协调型岗位、伪 AI 岗，还是技术落地型岗位？

4. `风险提示`
   我的短板在哪里，这个岗位的风险在哪里？

5. `行动建议`
   如果继续投，我现在应该怎么做？

### 5.2 情绪需求

这个项目不只解决信息需求，也解决情绪需求：

- 降低用户面对 JD 的无从判断感
- 降低“我是不是不够格”的空泛焦虑
- 给出更稳定、可执行的下一步

### 5.3 产品要求

输出必须满足：

- `直接`
- `可信`
- `可解释`
- `不羞辱用户`

## 6. 产品解法

### 6.1 输入

MVP 首版输入只保留 4 项：

1. `JD 文本`
2. `简历文本`
3. `用户当前状态`
4. `用户目标`

用户目标先支持 4 类：

- `求稳`
- `冲高薪`
- `转 AI`
- `找长期主线`

### 6.2 输出

MVP 首版输出固定为 6 个模块：

1. `投递建议`
   `冲 / 可投 / 谨慎 / 避开`

2. `总体匹配度`
   给出一个总分，但不把它做成唯一结论

3. `岗位类型`
   例如：`成长型 AI PM / 协调型 PM / 伪 AI 岗 / 技术落地型 PM / 增长型 AI PM`

4. `匹配证据`
   明确说明 JD 哪些要求与你已有经历能对应上

5. `风险提示`
   明确指出短板和岗位风险

6. `下一步建议`
   例如：先补案例、改简历、准备哪类面试表达

### 6.3 核心逻辑

MVP 的核心不是“大模型自由发挥”，而是：

`结构化判断框架 + LLM 生成解释`

即：

- 先把 JD 和简历拆成结构
- 再按预设维度打判断
- 最后用 LLM 把判断翻译成自然语言结果

这样可以降低输出飘忽不稳的问题。

## 7. 判断框架

### 7.1 岗位侧维度

先定义 6 个岗位维度：

1. `AI 技术密度`
2. `真实用户/业务接触度`
3. `系统落地深度`
4. `数据指标责任`
5. `商业结果关联`
6. `流程协调倾向`

### 7.2 候选人侧维度

再定义 7 个候选人维度：

1. `AI 项目经验`
2. `产品设计能力`
3. `场景抽象能力`
4. `技术理解能力`
5. `数据分析能力`
6. `业务结果表达`
7. `跨团队推动能力`

### 7.3 输出逻辑

初版逻辑可以简单处理：

- 先提取 JD 的高权重要求
- 再从简历里找对应证据
- 计算 `匹配程度`
- 单独判断 `岗位成长性` 与 `伪 AI 风险`
- 综合生成最终建议

### 7.4 投递建议规则

先用规则驱动，保持可控：

- `冲`
  匹配度高，岗位成长性高，核心短板少

- `可投`
  匹配度中高，有少量短板，但可通过表达或补材料解决

- `谨慎`
  匹配不稳定，或岗位本身存在明显消耗风险

- `避开`
  匹配度低，或岗位明显是伪 AI/高消耗低成长岗位

## 8. 游戏化与风格方向

### 8.1 角色定位

MVP 可以引入 `卡皮巴拉` 作为轻量陪伴角色，但不把它做成重 IP 产品。

角色定位：

`一只情绪稳定、见过很多 JD 的求职观察员`

### 8.2 游戏化原则

游戏化只做轻量增强，不干扰主判断：

- 降低焦虑感
- 让结果更愿意读完
- 增强分享感

### 8.3 具体落点

MVP 里游戏化只落在 4 个地方：

1. `欢迎与引导文案`
2. `分析过程中的状态反馈`
3. `结果页的轻人格表达`
4. `整体 UI 的柔和设计语言`

### 8.4 明确不做

- 不做复杂剧情
- 不做养成系统
- 不做积分商城
- 不做重动画

## 9. 功能范围

### 9.1 MVP 必做功能

1. 文本输入区
2. 用户身份与目标选择
3. 一键分析
4. 结果页展示
5. 再分析能力
6. 示例数据体验

### 9.2 可选但次优先级功能

- 保存历史分析结果
- 导出分析结果
- 分享卡片
- 简历优化建议展开层

### 9.3 当前不做功能

- 文件上传解析 PDF
- 登录系统
- 多用户数据库
- 招聘网站抓取
- 真正的简历重写器
- 自动投递

## 10. 用户流程

### 10.1 主流程

1. 用户进入首页
2. 看到工具定位与输入区
3. 粘贴 JD
4. 粘贴简历
5. 选择当前状态
6. 选择目标
7. 点击分析
8. 看到结果页
9. 根据建议继续修改后再次分析

### 10.2 结果页结构

结果页建议按以下顺序展示：

1. `一句话结论`
2. `投递建议标签`
3. `匹配度与岗位类型`
4. `你能打的点`
5. `会卡住的点`
6. `岗位风险`
7. `下一步动作`

这样先给结论，再展开解释。

## 11. 技术实现方案

### 11.1 技术目标

技术方案要满足：

- 单人能快速开发
- 本地即可跑通
- 后续容易接 API
- 前后端复杂度低

### 11.2 推荐架构

建议做成一个轻量 Web 应用：

- `Frontend`
  一个单页应用，负责输入、状态切换、结果展示

- `Backend`
  一个轻量 API 服务，负责：
  - 接收输入
  - 做文本预处理
  - 调用判断逻辑
  - 调用 LLM
  - 返回结果 JSON

### 11.3 技术栈建议

从单人开发效率出发，建议：

- `Frontend`: React + Vite + TypeScript
- `UI`: Tailwind CSS
- `Backend`: Python FastAPI
- `LLM 调用`: OpenAI API 或兼容接口
- `本地存储`: 先不做数据库，必要时用本地 JSON / SQLite

原因：

- 前端可以快速做出体验
- FastAPI 适合快速搭 API 和调试 prompt / rule logic
- Python 更适合后续做文本处理和规则判断

### 11.4 模块划分

后端建议拆成 5 个模块：

1. `input_parser`
   处理输入清洗

2. `jd_analyzer`
   分析 JD，提取岗位要求和岗位类型信号

3. `resume_analyzer`
   分析简历，提取候选人能力证据

4. `matcher`
   按规则做维度匹配和建议判断

5. `report_generator`
   生成用户最终看到的结果文案

### 11.5 前后端数据流

主流程如下：

1. 前端收集输入
2. 请求 `/analyze`
3. 后端清洗数据
4. 后端调用分析逻辑
5. 后端生成结构化结果
6. 后端调用 LLM 生成自然语言解释
7. 返回前端展示

### 11.6 API 设计草案

MVP 阶段先保留两个接口：

- `POST /analyze`
  输入：`jd_text`、`resume_text`、`user_level`、`goal`
  输出：完整分析结果

- `GET /demo`
  输出：一组示例 JD 和简历数据

## 12. LLM 使用策略

### 12.1 原则

LLM 不负责直接拍脑袋下结论。  
LLM 主要做两件事：

1. 从文本中提取结构化信息
2. 把结构化判断翻译成自然语言结果

### 12.2 初版调用方式

建议初版分两次调用：

1. `抽取阶段`
   分别抽取 JD 信号和简历证据

2. `解释阶段`
   根据规则结果生成最后的总结与建议

这样更稳定，也更容易 debug。

### 12.3 风险控制

为了降低幻觉和不稳定：

- 先给固定输出 schema
- 限制字段范围
- 尽量让模型引用输入中的真实证据
- 最终建议优先由规则决定，不让模型自己随意改结论

## 13. 开发阶段建议

### 13.1 第一阶段

先做 `无数据库、无登录、纯文本输入` 的跑通版本。

要点：

- 页面能用
- 分析链路能走通
- 输出结构稳定

### 13.2 第二阶段

再优化：

- 结果体验
- 文案语气
- 轻量卡皮巴拉形象
- 分享体验

### 13.3 第三阶段

如果验证有效，再考虑：

- 历史记录
- 文件上传
- 更细的岗位分类
- 更强的简历建议模块

## 14. 风险

MVP 主要风险有 5 个：

1. 判断逻辑太主观
2. LLM 输出不稳定
3. JD 与简历文本质量不齐
4. 输出像“空洞分析报告”
5. UI 太轻浮会损伤可信度

对应策略：

- 用结构化维度和规则兜底
- 限定 schema
- 提供示例输入
- 输出必须有证据和动作建议
- 游戏化只做轻量装饰

## 15. 当前结论

这个项目的 MVP 可以成立。

它最有机会打中的不是“简历优化”红海，而是一个更窄、更具体、也更真实的场景：

`用户想知道一个 AI PM 岗位值不值得投，以及自己该不该现在就投。`

如果首版能把这件事做准、做直接、做得不让人焦虑，它就已经具备发布和验证价值。
